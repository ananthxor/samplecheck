## Tech Debt

### Phase 14: Custom Reports

**ISSUE-01 — `resolution` field silently ignored on re-run (RPT-01)**
- **Impact:** User selects "hourly" resolution when creating a report, but re-running always fetches daily-grain data via `useAnalytics`. No error — just silent mismatch between saved config and displayed data.
- **File:** `apps/web/src/features/reports/pages/reports-page.tsx` lines 25-28
- **Fix:** Either wire `resolution = 'hourly'` to `useHourlyMetrics`, or remove the hourly option from the builder until wired.

**ISSUE-02 — Wasted `useAnalytics` query on `/reports` mount (RPT-04)**
- **Impact:** On initial page load, `useAnalytics` fires with `startDate = ''` and `endDate = ''` because the `enabled` guard only checks `profile?.advertiser_id`. Returns 0 rows harmlessly. UI is correct due to `safeReportData` guard.
- **File:** `apps/web/src/features/reports/pages/reports-page.tsx` line 27
- **Fix:** Add `&& !!startDate && !!endDate` to the `enabled` condition.

**ISSUE-03 — XLS report export shows UUIDs instead of creative/campaign names (RPT-04)**
- **Impact:** `exportReportXls(runningData, {}, {}, filename)` passes empty name maps. XLS "Creative" and "Campaign" columns show raw UUID strings.
- **File:** `apps/web/src/features/reports/components/saved-reports-list.tsx` line 178
- **Fix:** Fetch creative/campaign name maps from the API (similar to what the main analytics page does) and pass them to `exportReportXls`.
