**Critical pitfall — read this first:**
`pickCategory(id)` expects `cat.id` (e.g. `"cat-1"`), NOT `cat.key` (e.g. `"interactive"`). The `initialCategoryKey` prop uses the `key` field. The `useEffect` must do a two-step lookup: find the category by `key`, then pass its `id` to `pickCategory`.

**Changes to make in `creatives-selector.tsx`:**

1. Add `initialCategoryKey?: string` to the component props interface (or add it as a destructured prop if the component currently accepts no props).

2. The component signature currently is `export default function Creatives()`. Change it to:
   ```tsx
   interface CreativesProps {
     initialCategoryKey?: string;
   }

   export default function Creatives({ initialCategoryKey }: CreativesProps = {}) {
   ```

3. Add a `useEffect` that fires once on mount (empty dependency array `[]`). Inside it, if `initialCategoryKey` is truthy, find the matching category by `key` in `adCategories`, then call `pickCategory(cat.id)`:

   ```tsx
   // eslint-disable-next-line react-hooks/exhaustive-deps
   useEffect(() => {
     if (!initialCategoryKey) return;
     const cat = adCategories.find((c) => c.key === initialCategoryKey);
     if (cat) pickCategory(cat.id);
   }, []);
   ```

   Place this `useEffect` immediately after the existing state declarations, before any derived variables. The eslint-disable comment is required because `pickCategory` is defined in the same component scope and the linter would normally flag it as a missing dep — but we intentionally run this only once on mount.

4. Do NOT change any other logic in this file. The `pickCategory` function already handles setting the correct step (step 1 if `hasAdSizes`, step 2 if not), so the "skip to step 2 for Ad Size" behavior is automatic.

**Verify the slug-to-key mapping from fmtData.ts (do not guess — check the file):**
- `interactive` → category with `key: "interactive"` → `id: "cat-1"` (hasAdSizes: true → skips to step 1)
- `animated` → category with `key: "animated"` → `id: "cat-2"` (hasAdSizes: true → skips to step 1)
- `desktop` → category with `key: "desktop"` → `id: "cat-3"` (hasAdSizes: true → skips to step 1)
- `native` → category with `key: "native"` → `id: "cat-4"` (hasAdSizes: false → skips to step 2)
- `standard-banners` → category with `key: "standard-banners"` → `id: "cat-5"` (hasAdSizes: false → skips to step 2)

These keys are looked up dynamically from `adCategories` — no hardcoding of IDs needed in the component.

**The slug mapping table** (dashboard slugs → fmtData category keys):

```typescript
const SLUG_TO_CATEGORY_KEY: Record<string, string> = {
  interactive: 'interactive',
  animated: 'animated',
  desktop: 'desktop',
  standard: 'standard-banners',
  native: 'native',
};
```

Note: The dashboard "Video" card uses slug `video`, which has no matching fmtData category. When `SLUG_TO_CATEGORY_KEY[typeParam]` is `undefined`, `initialCategoryKey` is `undefined` and the browser opens at step 0 (Choose Ad Type) — this is the correct fallback behavior.

**Complete file after changes:**

```tsx
import { useSearchParams } from 'react-router'
import Creatives from './creatives-selector'

const SLUG_TO_CATEGORY_KEY: Record<string, string> = {
  interactive: 'interactive',
  animated: 'animated',
  desktop: 'desktop',
  standard: 'standard-banners',
  native: 'native',
};

export default function CreativesNewPage() {
  const [searchParams] = useSearchParams()
  const typeSlug = searchParams.get('type') ?? ''
  const initialCategoryKey = SLUG_TO_CATEGORY_KEY[typeSlug]

  return (
    <div className="mt-0">
      <Creatives initialCategoryKey={initialCategoryKey} />
    </div>
  )
}
```
