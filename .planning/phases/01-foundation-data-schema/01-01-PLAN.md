  <action>
    **Step 1: Install Supabase CLI and initialize**

    Install the Supabase CLI globally via npm:
    ```
    npm install -g supabase
    ```
    Then run `npx supabase init` in the project root to create the `supabase/` directory with `config.toml`.
    Log in with `npx supabase login` (this opens a browser for auth token -- if it fails, user provides the token).
    Link to the remote project: `npx supabase link --project-ref <PROJECT_REF>` using the reference ID from Task 1.

    **Step 2: Create .gitignore**

    Create `.gitignore` at project root with entries for:
    - `.env.*` (except `.env.example`)
    - `node_modules/`
    - `dist/`
    - `.supabase/` (local Supabase data)

    **Step 3: Create environment files**

    Create `.env.example` with placeholder keys:
    ```
    VITE_SUPABASE_URL=your_supabase_project_url
    VITE_SUPABASE_ANON_KEY=your_supabase_anon_key
    SUPABASE_SERVICE_ROLE_KEY=your_service_role_key_server_only
    SUPABASE_DB_PASSWORD=your_database_password
    ```

    Create `.env.development` with the actual values from Task 1. NEVER commit this file.
    Create `.env.production` as a placeholder with empty values (prod project created later).

    **Step 4: Write the initial schema migration**

    Create `supabase/migrations/20260219000000_initial_schema.sql` with the COMPLETE schema from the research document. This includes:

    1. **Custom enums**: `event_type` (10 values: impression_served, impression_viewable, engagement, click, video_play, video_pause, video_complete, expand, collapse, close), `user_role` (super_admin, advertiser), `creative_status` (draft, active, paused, archived), `campaign_status` (draft, active, paused, completed)

    2. **Tables** (all with UUID PKs via gen_random_uuid(), timestamps):
       - `advertisers`: id, name, contact_email, created_at, updated_at
       - `user_profiles`: id, auth_user_id (FK to auth.users), advertiser_id (FK to advertisers), role (user_role enum), display_name, must_change_password (boolean, default true), created_at, updated_at. UNIQUE on auth_user_id.
       - `campaigns`: id, advertiser_id (FK to advertisers, CASCADE), name, status (campaign_status), created_at, updated_at
       - `creatives`: id, advertiser_id (FK to advertisers, CASCADE), campaign_id (FK to campaigns, SET NULL), name, format_id (TEXT), format_name (TEXT), status (creative_status), schema_version (INTEGER DEFAULT 1), template_data (JSONB DEFAULT '{}'), thumbnail_url (TEXT), width (INTEGER), height (INTEGER), created_at, updated_at
       - `ad_events`: id (UUID), event_timestamp (TIMESTAMPTZ, NOT NULL, DEFAULT now()), event_type (event_type enum), request_id (UUID NOT NULL), creative_id (UUID NOT NULL), campaign_id (UUID), advertiser_id (UUID NOT NULL), extra_data (JSONB DEFAULT '{}'), created_at (TIMESTAMPTZ). **PRIMARY KEY (event_timestamp, id)**. **PARTITION BY RANGE (event_timestamp)**.
       - `daily_metrics`: id, metric_date (DATE), advertiser_id (FK), campaign_id (FK), creative_id (FK), impressions_served (BIGINT), impressions_viewable, clicks, engagements, video_plays, video_completes, total_dwell_time_ms (all BIGINT DEFAULT 0), created_at, updated_at. UNIQUE on (metric_date, advertiser_id, campaign_id, creative_id).

    3. **Partitions**: Create 6 monthly partitions for ad_events: 2026_02 through 2026_07.

    4. **Indexes**: On ad_events: request_id, creative_id, advertiser_id, event_type. On creatives: advertiser_id, campaign_id. On campaigns: advertiser_id. On daily_metrics: (advertiser_id, metric_date), (campaign_id, metric_date). On user_profiles: auth_user_id, advertiser_id.

    5. **Helper functions**:
       - `get_user_advertiser_id()`: Returns UUID, STABLE, SECURITY DEFINER. Queries user_profiles where auth_user_id = (SELECT auth.uid()).
       - `is_super_admin()`: Returns BOOLEAN, STABLE, SECURITY DEFINER. Checks user_profiles for role = 'super_admin'.

    6. **RLS policies** (enable RLS on ALL tables):
       - `advertisers`: Super admin full access (ALL), users SELECT own (via get_user_advertiser_id())
       - `user_profiles`: Super admin full access (ALL), users SELECT own (via auth.uid()), users UPDATE own
       - `campaigns`: Super admin full access (ALL), users manage own (ALL, via get_user_advertiser_id())
       - `creatives`: Super admin full access (ALL), users manage own (ALL, via get_user_advertiser_id())
       - `ad_events`: Super admin SELECT all, users SELECT own (via get_user_advertiser_id()). No INSERT/UPDATE/DELETE for users -- events inserted via service_role from Edge Functions.
       - `daily_metrics`: Super admin full access (ALL), users SELECT own

    7. **Triggers**: `handle_updated_at()` function and triggers on advertisers, user_profiles, campaigns, creatives, daily_metrics.

    **IMPORTANT RLS performance**: Always use `(SELECT auth.uid())` and `(SELECT public.is_super_admin())` with the SELECT wrapper for caching.

    **Step 5: Write the storage bucket migration**

    Create `supabase/migrations/20260219000001_storage_bucket.sql`:
    - Insert into storage.buckets: id='creative-assets', name='creative-assets', public=true, file_size_limit=52428800 (50MB), allowed_mime_types for images (png, jpeg, gif, webp, svg+xml), video (mp4, webm), and code (javascript, html, css).
    - Storage RLS policies:
      - Authenticated users can INSERT to own folder (bucket_id check + foldername[1] = advertiser_id)
      - Public SELECT access (bucket is public)
      - Authenticated users can UPDATE own files
      - Authenticated users can DELETE own files

    **Step 6: Write seed data**

    Create `supabase/seed.sql` with test data for dev:
    - One test advertiser: name='Test Advertiser', contact_email='test@scrolltoday.dev'
    - One test campaign: name='Test Campaign', status='draft', linked to test advertiser
    - One test creative: name='Test Creative', format_id='carousel', format_name='Swipeable Carousel', status='draft', schema_version=1, linked to test advertiser and campaign
    - One test ad_event insert to verify partitioning works (use current month's timestamp)
    - Comment explaining that seed data should be applied manually: `npx supabase db reset --linked` or via SQL editor

    **Step 7: Push migrations to Supabase**

    Run: `npx supabase db push`
    This applies all migrations in order to the linked remote project.

    If db push fails due to network or permissions, troubleshoot by checking:
    - `npx supabase projects list` (confirms CLI is logged in)
    - `npx supabase db push --dry-run` (previews what would be applied)

    **Step 8: Generate TypeScript types**

    Run: `npx supabase gen types typescript --project-id <PROJECT_REF> --schema public > database.types.ts`
    Place the generated file temporarily at project root -- it will be moved to packages/shared/src/ in Plan 02.
    This verifies the schema is queryable and produces the type file needed by the frontend.
